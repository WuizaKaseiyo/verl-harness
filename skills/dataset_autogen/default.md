dataset_autogen skill — author a verl-compatible preprocess script for a HuggingFace dataset that isn't in `dataset_registry`.

## Pre-requisites

Before generating a script:

1. The user gave a HuggingFace dataset id (e.g., `myorg/mydata`). It must resolve — `datasets.load_dataset(id, split='train', streaming=True)` succeeds.
2. The trainer type from `training_intent.md` is known (`sft`, `ppo`, `grpo`, `dpo`/`reward_model`, etc.). It drives which column convention the output parquet must match.
3. The dataset's *shape* is identifiable from a single streaming example:
   - single-turn (prompt → response)
   - multi-turn chat (list of turns)
   - multi-turn with tool calls (turns may include `tool_calls`)
   - reward-model preference pairs (`chosen` / `rejected`)
   - math/QA with ground-truth answer
   - multi-choice (A/B/C/D)

## Template selection

Match the (trainer-type, shape) to one of verl's existing preprocess scripts as a template. The chosen template's *structure* is copied verbatim; only the dataset-loading and field-extraction logic is rewritten to match the new dataset's schema.

| Trainer | Shape | Template (under `examples/data_preprocess/`) |
|---|---|---|
| SFT | single-turn | `gsm8k.py` — straightforward `prompt` + `response` extraction |
| SFT | multi-turn | `multiturn.py` — preserves conversation turns |
| SFT | multi-turn + tool | `gsm8k_multiturn_sft.py` — preserves tool calls in chat |
| PPO/GRPO | single-turn math | `math_dataset.py` — rule-based reward via ground truth |
| PPO/GRPO | single-turn QA | `gsm8k.py` |
| PPO/GRPO | multi-turn + tool | `gsm8k_multiturn_w_tool.py` |
| PPO/GRPO | multi-choice | `hellaswag.py` |
| DAPO multi-turn | tool | `dapo_multiturn_w_tool.py` |
| Reward model | preference pairs | `full_hh_rlhf.py` |
| Tool-agent loop | any | `gsm8k_tool_agent_loop.py` |

## Field-mapping protocol

For each verl-expected column, decide *where it comes from* in the HF dataset:

- `prompt` (always list of `{role, content}`):
  - If the HF row has a `messages` field → use it (verify roles ∈ {user, assistant, system}).
  - Else if it has `question` / `instruction` / `prompt` → wrap as `[{"role":"user","content":<that>}]`.
  - Else if it has nested chat fields (`conversation`, `dialog`) → flatten.

- `data_source` (string):
  - Use the HF dataset id verbatim (e.g., `"openai/gsm8k"`).

- `ability` (string):
  - Infer from dataset card if obvious (`"math"`, `"qa"`, `"reasoning"`, `"code"`, `"multi-choice"`, etc.). Record the inference in the rationale file.

- `reward_model.style` (string):
  - For PPO/GRPO with a deterministic checker (math, multi-choice): `"rule"`. The preprocess script must extract the ground truth into `reward_model.ground_truth`.
  - For preference-based reward models: `"model"`. Ground-truth is the dispreferred completion (used as a contrastive signal).

- `reward_model.ground_truth` (string):
  - Math: the final numerical answer (regex out of the dataset's solution field).
  - Multi-choice: the correct letter / index.
  - Free-form: leave empty and document in `reward_model.style` that the run depends on a learned reward model (which is then the user's responsibility — note this clearly).

- `response` (SFT only):
  - HF row's `response` / `answer` / `output` / `target`, whichever maps to the assistant's intended completion.

- `chosen` / `rejected` (reward-model only):
  - HF row's `chosen` / `rejected` fields directly. Validate they're list-of-dicts already; if they're strings, wrap them in `[{"role":"assistant","content":<str>}]`.

```python
            "data_source": "<hf id>",
            "prompt": prompt,
            "ability": "<inferred>",
            "reward_model": {"style": "rule", "ground_truth": ground_truth},
            "extra_info": {"split": split, "index": idx},
        }
    return process_fn


if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--local_dir", required=True)
    parser.add_argument("--hdfs_dir", default=None)
    args = parser.parse_args()

    ds = datasets.load_dataset("<hf id>")

    train = ds["train"].map(function=make_map_fn("train"), with_indices=True)
    test = ds.get("test") or ds.get("validation")
    if test is not None:

    os.makedirs(args.local_dir, exist_ok=True)
    train.to_parquet(os.path.join(args.local_dir, "train.parquet"))
    if test is not None:
        test.to_parquet(os.path.join(args.local_dir, "test.parquet"))
```

Adapt `process_fn` for the trainer type (single-turn vs multi-turn vs preference pair).

## Rationale file

Alongside `preprocess.py`, write `workspace/dataset/<name>/preprocess_rationale.md`:

```markdown
# Preprocess generation rationale

## HF dataset
- id: myorg/mydata
- splits: train (12345 rows), test (678 rows)
- sample row: { "question": "...", "answer": "..." }

## Trainer / shape detected
- trainer: grpo (from training_intent.md)
- shape: single-turn math-style (question + numerical answer)

## Template chosen
- examples/data_preprocess/math_dataset.py
  - Reason: closest match for single-turn math with rule-based reward.

## Field mapping
- prompt ← [{"role":"user","content": example["question"]}]
- data_source ← "myorg/mydata"
- ability ← "math"
- reward_model.style ← "rule"
- reward_model.ground_truth ← extract_final_number(example["answer"])
- extra_info ← {"split": split, "index": idx}

## Decisions
- Ground-truth extraction uses the regex `#### (-?\d+\.?\d*)`, mirroring gsm8k.
- No `system` prompt added — the user's intent is plain answer extraction.

## Risks
- The `answer` field contains free-form reasoning in some rows. The regex extracts only the final number; rows where no number appears will have an empty ground-truth, which the verl rule-checker will treat as "no match". This is logged but does not block.
```

## Self-check before handing off

The generated script must:

- Parse: `python -m py_compile workspace/dataset/<name>/preprocess.py`.
- Run on a 100-row subset without errors (use `streaming=True` + `.take(100)`).
- Produce parquet rows whose column set is a superset of the trainer-required columns (see `dataset_registry` column conventions).

If any check fails, **regenerate** — do not patch around the failure.

## Things you must not do

- Do not invent fields. Every verl-expected column must come from a real HF field or a deterministic function of HF fields. If a required field cannot be derived, the dataset is unsuitable for the chosen trainer — report it.
- Do not silently drop rows. If many rows fail field extraction, that's a dataset-shape mismatch worth flagging in the rationale; do not just filter and pretend everything is fine.
- Do not author preprocess scripts for *unsupported* trainer shapes. If the user picked a trainer this skill has no template for, halt and ask.
